# マシン・ペイメント・プロトコル (MPP) ブリーフィング・ドキュメント

## エグゼクティブ・サマリー

マシン・ペイメント・プロトコル（MPP）は、TempoとStripeが共同で策定した、インターネット・ネイティブなマシン間決済（M2M）のためのオープン標準プロトコルです。既存の金融システムは人間が操作することを前提に設計されており、自律型エージェントにとってはアカウント作成や複雑なチェックアウトフローが障害となっていました。MPPは、HTTP 402「Payment Required」ステータスコードを標準化し、エージェントやアプリが単一のHTTPリクエスト内でサービスの発見、支払い、利用をプログラムによって完結できる仕組みを提供します。

MPPは決済方法に依存せず、ステーブルコイン（Tempo）、クレジットカードや法定通貨（Stripe）、ビットコイン（Lightning）など、多様な決済レールを単一のインターフェースで統合可能です。また、即時決済の「charge」インテントに加え、高スループットかつ低コストな従量課金を実現する「session」インテントを備えており、LLMのトークン課金やストリーミング型のサービスに最適化されています。

--------------------------------------------------------------------------------

## 1. プロトコルの背景と目的

### 既存決済システムの課題

現在のインターネット決済は、ブラウザ自動化の防止（CAPTCHAなど）や、人間による手動入力を前提とした複雑なフォームに依存しています。これらはプログラムによる消費において、信頼性の低下、レイテンシの増加、コストの高騰を招く「インターフェース・レベルのグローバルな問題」となっています。

### MPPの役割

MPPは、買い手と売り手の間でコストや決済方法を交渉するためのプログラム向けインターフェースを提供します。

- **自律性の付与:** エージェントがOAuthや手動のアカウント設定なしで、自律的にサービスへ支払うことを可能にします。
- **摩擦の排除:** サービスプロバイダーは、複雑なオンボーディングや請求管理を介さず、APIリクエストごとに直接収益化を図れます。

--------------------------------------------------------------------------------

## 2. コア・メカニズムと支払いフロー

MPPの通信は、チャレンジ（Challenge）、クレデンシャル（Credential）、レシート（Receipt）の3つのフェーズで構成されます。

### 基本フロー

1. **リクエスト:** クライアントが有料リソースを要求する（`GET /resource`）。
2. **チャレンジ:** サーバーが **402 Payment Required** を返し、`WWW-Authenticate: Payment` ヘッダーを通じて支払い要件（金額、通貨、受け取り先など）を提示する。
3. **支払い履行:** クライアントが提示された決済方法を選択し、支払いを実行する。
4. **クレデンシャル送付:** クライアントが支払い証明（Credential）を `Authorization: Payment` ヘッダーに含めてリクエストを再試行する。
5. **検証とリソース提供:** サーバーがクレデンシャルを検証・決済し、**200 OK** と共にリソースおよび配信証明となる **レシート（Receipt）** を返却する。

### 主要な構成要素

|   |   |   |
|---|---|---|
|要素|役割|内容|
|**Challenge**|サーバーによる支払い要求|チャレンジID、金額、通貨、決済方法の詳細、有効期限。|
|**Credential**|クライアントによる支払い証明|チャレンジIDへの参照、支払者の識別情報、決済方法固有の証明データ。|
|**Receipt**|サーバーによる受領確認|決済成功のステータス、参照ID（トランザクションハッシュ等）、決済時刻。|

--------------------------------------------------------------------------------

## 3. 決済インテント（支払い形態）

MPPでは、利用シーンに合わせて「charge」と「session」の2つのインテントを定義しています。

### インテントの比較

|   |   |   |
|---|---|---|
|特徴|Charge (一括決済)|Session (セッション決済)|
|**パターン**|リクエストごとの即時決済|プリペイド預金による継続的な従量課金|
|**オーバーヘッド**|決済ネットワークへの通信が発生|オフチェーン署名検証によりほぼゼロ|
|**スループット**|1リクエストにつき1決済|1セッションで数百〜数千リクエスト|
|**最適な用途**|単発のAPI実行、コンテンツ購入|LLMトークン、メタデータ、ストリーミング|
|**決済タイミング**|即時（オンチェーン等）|預金後、随時差し引き、終了時に精算|

--------------------------------------------------------------------------------

## 4. サポートされる決済メソッド

MPPは決済レールを選ばない「アグノスティック」な設計です。現在、以下のメソッドが本番環境で稼働しています。

- **Tempo (ステーブルコイン):**
    - Tempoブロックチェーン上のTIP-20トークンを使用。
    - 約500ミリ秒の確定的なファイナリティと、0.1セント以下の低コストな手数料が特徴。
- **Stripe (カード・法定通貨):**
    - 共有支払いトークン（SPT）を使用し、既存のカードインフラで決済。
    - Stripeの税務計算、不正防止、レポート機能をそのまま利用可能。
- **Lightning (ビットコイン):**
    - Lightning Network上のBOLT11インボイスを使用。
    - 暗号学的な検証がサーバーローカルで完結し、即時決済が可能。
- **Card (暗号化ネットワークトークン):**
    - Visa等のカードネットワークが提供する単回利用のトークンを使用。

--------------------------------------------------------------------------------

## 5. 技術的特徴とセキュリティ

### パフォーマンスと拡張性

- **セッションの効率性:** セッション・インテントでは個別のリクエストにコンセンサスを必要とせず、CPUの処理能力がボトルネックになるほどの高速なスループット（2桁ミリ秒のレイテンシ）を実現します。
- **マルチメソッド対応:** 単一のエンドポイントでTempo、Stripe、Lightningを同時に受け入れることができ、クライアントは自身がサポートするレールを選択できます。

### セキュリティの原則

- **TLS 1.2+ 必須:** すべての通信には暗号化が強制されます。
- **リプレイ保護:** クレデンシャルは単回利用（Single-use）であり、再利用は拒否されます。
- **アイデムポテンシー（冪等性）:** 支払われていないリクエストに対してサーバー側で副作用（データベース書き込み等）を発生させないことが求められます。
- **バインディング:** チャレンジIDはパラメータと暗号学的に紐付けられ、内容の改ざんを防止します。

--------------------------------------------------------------------------------

## 6. 開発エコシステム

MPPは開発者の導入を容易にするためのツール群を提供しています。

- **公式SDK:** TypeScript (`mppx`), Python (`pympp`), Rust (`mpp-rs`) が提供されています。
- **LLMとの親和性:** `llms-full.txt` を通じて、コーディング・エージェントにプロトコルの全コンテキストを直接読み込ませることが可能です。
- **MCP (Model Context Protocol) 統合:** MCPサーバーがツールの呼び出しを直接収益化でき、エージェントはOAuthなしで自動的に支払いを実行できます。
- **フレームワーク対応:** Hono, Express, Next.js, Elysia 等の主要なフレームワーク用ミドルウェアが提供されており、数行のコードで決済ゲート付きAPIを構築できます。

Machine Payments Protocol（MPP）とは何ですか？

**Machine Payments Protocol（MPP）**は、**マシン間（M2M）の支払いを実現するためのオープンな標準プロトコル**です。AIエージェント、アプリケーション、あるいは人間が、**同一のHTTPリクエスト内でサービスに対する支払い**を行えるように設計されています。

MPPの主な特徴と仕組みは以下の通りです。

1. 概要と目的

- **開発者:** **TempoとStripe**によって共同執筆されました。
- **標準化:** IETF（Internet Engineering Steering Group）に「Payment HTTP Authentication Scheme」として提案されているオープン規格です。
- **解決する課題:** 従来の決済システムは人間向けに構築されており、アカウント作成やUI操作、キャプチャ認証などが必要で、自律的なエージェントには不向きでした。MPPはこれらを排除し、**プログラムによる自動決済**を可能にします。

2. 技術的な仕組み（HTTP 402の標準化）

MPPは、これまであまり活用されてこなかった**HTTPステータスコード「402 Payment Required（支払いが必要）」を標準化**しています。基本的なフローは以下の通りです。

1. **Challenge (402):** クライアントがリソースを要求すると、サーバーは402エラーを返し、金額や通貨、支払い方法を記した「チャレンジ」を提示します。
2. **Credential:** クライアントはチャレンジに従って支払いを行い、支払い証明となる「クレデンシャル」を付けてリクエストを再試行します。
3. **Receipt:** サーバーは支払いを検証し、リソースと共に「レシート（受領証）」を返します。

4. 多様な支払い方法と柔軟性

MPPは特定の決済手段に依存しない**アグノスティック（不可知）なプロトコル**です。

- **サポートされる決済:** ステーブルコイン（Tempo）、クレジットカードや銀行振込（Stripe）、ビットコイン（Lightning Network）などが既に利用可能です。
- **通貨:** USD、EURなどの法定通貨から、USDC、BTCなどの暗号資産まで対応しています。

4. 支払い形態（インテント）

用途に合わせて、主に2つの支払い形態が定義されています。

- **Charge（チャージ）:** 1回のリクエストに対して1回支払う、**即時決済の単発支払い**です。
- **Session（セッション）:** 事前にデポジットを行い、利用量に応じてオフチェーンのバウチャーで支払う**従量課金方式**です。LLMのトークン生成ごとの課金など、**超低遅延・高頻度な支払い**に適しています。

5. HTTP以外の展開

HTTPだけでなく、AIエージェントがツールを利用するための**Model Context Protocol (MCP)**上でも動作します。これにより、エージェントはOAuthなどの複雑な認証なしで、自律的に有料ツールを呼び出し、支払いを行うことが可能になります。

StripeやLightningでの支払い方法を知りたい

Machine Payments Protocol（MPP）において、**Stripe**と**Lightning**を使用した支払い方法は、それぞれのネットワーク特性を活かした仕組みになっています。

1. Stripeによる支払い

Stripeを利用した支払いは、**Shared Payment Tokens (SPTs)** という仕組みに基づいています。これにより、実際の支払い詳細（カード番号など）を直接共有することなく、安全に取引を行えます。

- **仕組み:** クライアントとサーバーの両方がStripeアカウントを必要とします。クライアントがStripeのAPIやStripe.jsを使用してSPTを作成し、サーバーがそのトークンを消費して`PaymentIntent`を作成することで決済が完了します。
- **主なステップ:**
    1. **402 Challenge:** サーバーが金額やStripeのビジネスプロファイル情報を含むチャレンジを返します。
    2. **SPT作成:** クライアントはStripe Elementsや保存済みの支払い方法を使用して支払い情報を収集し、チャレンジ内容に合わせたSPTを発行します。
    3. **Credential送信:** クライアントは発行されたSPTを「クレデンシャル（支払い証明）」としてサーバーに送ります。
    4. **決済完了:** サーバーはSPTを使用してStripe側で決済を確定させ、リソースとレシートを返します。
- **サポート内容:** クレジットカード、デジタルウォレット、およびStripeがサポートするその他の支払い方法を利用可能です。

2. Lightning Networkによる支払い

Lightningを利用した支払いは、ビットコインの高速な決済能力を活用しており、**単発支払い（Charge）**と**従量課金（Session）**の2つの形態があります。

Lightning Charge（単発支払い）

- **仕組み:** サーバーがリクエストごとに新しい**BOLT11インボイス**を発行します。
- **ステップ:** クライアントがインボイスを支払い、決済完了時に判明する**プリイメージ（支払い完了の秘密鍵）**をクレデンシャルとしてサーバーに提示します。サーバーは、そのプリイメージのハッシュ値が元のインボイスと一致するかをローカルで検証します。

Lightning Session（従量課金・セッション支払い）

- **仕組み:** LLMのトークン生成のような高頻度な支払いに適しています。
- **ステップ:**
    1. **デポジット:** クライアントが最初にまとまった金額（デポジット）を支払い、その際のプリイメージをベアラートークン（持参人トークン）として使用します。
    2. **継続利用:** 以降の各リクエストでは、このプリイメージを提示するだけで、サーバーが残高から利用分を差し引いていきます。
    3. **払い戻し:** セッション終了時、未使用の残高はクライアントが提示した返金用インボイスを通じて払い戻されます。

Lightningの実装には、リファレンスとして**Spark**ウォレットが使用されていますが、BOLT11インボイスの作成とプリイメージの検証ができる任意のLightningノードで動作可能です。

StripeのShared Payment Tokensの仕組みをもっと詳しく

Stripeの**Shared Payment Tokens (SPT)**は、Machine Payments Protocol (MPP) において、クライアント（AIエージェントなど）がサーバー（サービス提供者）に対して、実際のカード情報などを直接渡さずに安全に支払い手段を共有するための仕組みです。

その詳細な仕組みは以下の通りです。

1. SPTの基本的な役割

SPTは、顧客の支払い方法を特定のサーバーのStripeアカウントと共有するための「引換券」のような役割を果たします。

- **限定的な可視性:** サーバー側は、カードブランドや下4桁といった基本的な情報は確認できますが、実際の詳細な支払い情報（カード番号全体など）にはアクセスできません。
- **利用制限:** 有効期限や最大金額などの利用制限をSPTに設定することができます。

2. SPTを使用した決済の流れ

SPTによる支払いは、以下のステップで行われます：

1. **Challenge (サーバー):** サーバーがリソース要求に対し、`402 Payment Required`とともに「チャレンジ」を返します。これにはStripeのビジネスネットワークプロファイルや許可される支払い方法（カードなど）が含まれます。
2. **SPT作成 (クライアント):** クライアントはStripe Elementsや保存済みの支払い方法を使用して情報を収集し、**Stripe APIを呼び出してSPTを作成**します。この際、チャレンジの内容に合わせて使用制限を設定します。
3. **Credential送信 (クライアント):** クライアントは、生成されたSPT（`spt_...`で始まる文字列）を「クレデンシャル（支払い証明）」としてサーバーに送ります。
4. **決済実行 (サーバー):** サーバーは受け取ったSPTを使用して、自身のStripeアカウントで**PaymentIntentを作成・確定**させます。
5. **Receipt (サーバー):** 決済が完了すると、サーバーはリソースと共にレシートをクライアントに返します。

6. 技術的な要件と実装のポイント

- **双方のアカウント:** この仕組みを利用するには、**クライアントとサーバーの両方がStripeアカウントを保持している必要**があります。
- **トークン作成のプロキシ:** クライアント側でSPTを作成するにはStripeのシークレットキーが必要になるため、通常はクライアント自身のバックエンドサーバーにプロキシエンドポイントを作成し、そこを経由してStripe API（本番環境では`issued_tokens` API）を呼び出します。
- **Stripeインフラの活用:** SPT経由の支払いは、Stripeのダッシュボード上で通常の取引として表示されます。そのため、税金計算、不正防止（Stripe Radar）、レポート作成、返金処理などの既存のStripeインフラをそのまま利用できるという利点があります。

この仕組みにより、AIエージェントは人間が介入することなく、セキュアにStripeネットワークを通じたカード決済やウォレット決済を自動で行うことが可能になります。

AIエージェントが自律的に決済を行う仕組みを教えてください。

AIエージェントが人間を介さずに自律的に決済を行う仕組みは、主に**Machine Payments Protocol (MPP)**というオープンな標準プロトコルによって実現されています。

従来の決済システムは「人間」がUIを操作し、キャプチャ認証やアカウント作成を行うことを前提に設計されていましたが、MPPはこれらを排除し、**プログラム（マシン）がHTTPリクエスト内で直接交渉・支払いを行える**ようにしています。

主な仕組みは以下の通りです。

1. HTTP 402ステータスコードによる標準フロー

AIエージェントとサーバーの間では、以下の「チャレンジ・クレデンシャル・レシート」のフローが自動で実行されます。

- **チャレンジ (402 Payment Required):** エージェントが有料リソース（LLMのAPIなど）を要求すると、サーバーは402エラーと共に、金額や支払い方法を指定した「チャレンジ」を返します。
- **クレデンシャル (Credential):** エージェントは自身のウォレットを使用して支払いを実行、または承認し、その証明（クレデンシャル）をリクエストに付けて再試行します。
- **レシート (Receipt):** サーバーは支払いを検証し、リソースと共に支払完了証（レシート）を返します。

2. AIツールの呼び出し（MCPとの統合）

AIエージェントがツール（検索やデータ分析など）を利用するための標準規格である**Model Context Protocol (MCP)**上でも、この決済フローが動作します。 これにより、エージェントはOAuthなどの複雑な認証や事前のサブスクリプション契約なしで、**呼び出したツールに対してその場で自律的に支払い**を行うことが可能になります。

3. 高頻度な課金に対応する「セッション」の仕組み

LLMのトークン生成ごとの課金のように、非常に高頻度かつ低遅延な支払いが求められる場合、**「Session（セッション）」**という仕組みが使われます。

- エージェントは最初に対価をデポジット（預託）し、ペイメントチャネルを開設します。
- 以降の各リクエストでは、オフチェーンで署名された「バウチャー」を提示するだけで済みます。
- これにより、ブロックチェーンの処理速度に依存せず、マイクロ秒単位の高速な検証と、トークン単位の極小額決済（マイクロトランザクション）が実現されます。

4. 自律的な予算管理（Spend Controls）

エージェントが際限なく資金を使わないよう、**Tempo Wallet**などのエージェント向けウォレットには、**利用額制限（Spend Controls）**の機能が組み込まれています。これにより、開発者はエージェントに一定の予算を与え、その範囲内で自律的にサービスを購入させることができます。

5. 多様な支払い手段のサポート

エージェントは以下のネットワークを通じて支払いを行います。

- **Tempo:** ステーブルコイン（USDCなど）を使用し、約0.5秒での即時決済が可能です。
- **Stripe:** 共有支払いトークン（SPT）を使い、法定通貨やクレジットカードでの決済を自動化します。
- **Lightning Network:** ビットコインを使用し、低コストかつグローバルな決済を行います。

この仕組みにより、AIエージェントは「自らサービスを発見し、支払い、利用する」という完全な経済的自律性を持つことができるようになります。

一括払いとセッション型の支払いの違いは何ですか？

Machine Payments Protocol（MPP）における**一括払い（Charge）**と**セッション型支払い（Session）**の主な違いは、支払いの頻度、遅延（レイテンシ）、およびコスト効率にあります。

主な違いをまとめると以下の通りです。

1. 基本的な仕組みと支払いパターン

- **一括払い (Charge):** 1回のリクエストに対して固定額を即座に支払う、最もシンプルなパターンです。**「1リクエスト、1支払い、1レシート」**で完結します。
- **セッション型 (Session):** 高頻度な**従量課金（Pay-as-you-go）**を実現するための仕組みです。最初にオンチェーンの「エスクロー（預かり金）」に資金をデポジットしてペイメントチャネルを開設し、その後の各リクエストはオフチェーンの「バウチャー（署名済みの証明書）」を提示することで支払いを行います。

2. パフォーマンスと遅延（レイテンシ）

- **一括払い:** リクエストごとにインボイスの発行やネットワーク上での決済確定を待つ必要があるため、決済ネットワークの処理速度に依存した遅延が発生します。
- **セッション型:** チャネル開設後は、サーバー側でCPUによる署名検証（ecrecoverなど）を行うだけなので、**マイクロ秒単位の超低遅延**で処理されます。ブロックチェーンの処理能力（TPS）に制限されず、水平方向にスケール可能です。

3. コスト効率

- **一括払い:** 支払いのたびにオンチェーンでの取引が発生するため、リクエストごとに手数料がかかります。
- **セッション型:** 多数のオフチェーンバウチャーをまとめて1回のオンチェーン取引で精算するため、**1リクエストあたりの手数料をほぼゼロ**に抑えることができ、極小額（例：0.0001 USD）の課金も実用的になります。

4. 主なユースケース

|特徴|一括払い (Charge)|セッション型 (Session)|
|---|---|---|
|**最適な用途**|単発のAPI呼び出し、記事の購入、1回限りのダウンロード|**LLMのトークン課金**、ストリーミング、高頻度なAPI利用|
|**費用の確定**|事前に支払額が分かっている場合に適している|合計利用量が事前には分からない従量制に適している|

5. 精算と返金の仕組み

- **一括払い:** リクエストごとに即座に決済が確定（セットルメント）します。
- **セッション型:** セッション終了時に未使用のデポジット（残高）がクライアントに返金されます。サーバーは蓄積されたバウチャーを定期的に、またはセッション終了時にまとめてオンチェーンで精算します。

このように、**「シンプルで分かりやすい単発決済」の一括払い**に対し、**「高速・安価で高頻度なやり取り」を可能にするセッション型**という使い分けがなされています。

具体的な設定方法と仕組みは以下の通りです。

1. 設定の主な流れ

2. **サーバーのチャレンジ:** サーバーが`402 Payment Required`レスポンスを返す際、リクエストされたリソースに必要な金額（`amount`）や通貨（`currency`）などの詳細情報をチャレンジとして提示します。
3. **クライアントのコールバック:** クライアント側のSDK（`mppx`など）は、このチャレンジを受け取ると`createToken`コールバックを実行します。この際、サーバーが指定した金額や有効期限が引数（`params`）として渡されます。

- **金額制限 (****usage_limits[max_amount]****):** そのトークンで支払える最大金額を指定します。
- **通貨制限 (****usage_limits[currency]****):** トークンが使用される通貨（例: `usd`）を指定します。
- **有効期限 (****usage_limits[expires_at]****):** SPTが失効する日時を**Unixタイムスタンプ（秒）**で指定します。

```

  'usage_limits[max_amount]': amount,       // 最大金額を指定
  'usage_limits[expires_at]': expiresAt,    // 有効期限をタイムスタンプで指定
});

// Stripe APIを呼び出してSPTを作成
const response = await fetch('https://api.stripe.com/v1/.../tokens', {
  method: 'POST',
  headers: { /* Authorization 等 */ },
  body,
});
```

一括払いとセッション型を併用するメリットは？

Machine Payments Protocol（MPP）において、**一括払い（Charge）**と**セッション型（Session）**を併用することで、サービスの柔軟性と効率性を最大化できるというメリットがあります。

主なメリットは以下の通りです。

1. 利用シーンに合わせた最適な課金モデルの提供

サービス提供者（サーバー）は、単一のエンドポイントで両方の支払い方式を提示し、クライアントに選択させることができます。

- **一括払い（Charge）:** 1回限りの画像生成、記事の購入、固定価格のツール呼び出しなど、**「1リクエスト = 1支払い」**で済むシンプルな用途に適しています。
- **セッション型（Session）:** LLMによるトークン生成、ストリーミング、高頻度なデータ取得など、**「利用量が事前には不明で、超低遅延な支払い」**が求められる用途に適しています。

2. パフォーマンスとコストの最適化

両方をサポートすることで、取引の性質に応じて遅延と手数料を最小化できます。

- **遅延（レイテンシ）の解消:** セッション型を併用すれば、チャネル開設後はオフチェーンの「バウチャー」で支払うため、リクエストごとの決済ネットワーク待ち（数百ミリ秒〜数秒）がなくなります。検証はマイクロ秒単位で完了します。
- **マイクロトランザクションの実現:** 一括払いでは手数料負けしてしまうような極小額（例：0.0001 USD単位）の課金でも、セッション型なら多数の支払いを1回にまとめて精算するため、安価に提供可能になります。

3. クライアントの利便性と互換性の向上

クライアント（AIエージェントやアプリ）の状態や能力に合わせて、支払手段を柔軟に選ばせることができます。

- **状態管理の有無:** チャネルの開設や預託金（デポジット）の管理をしたくないシンプルなクライアントには「一括払い」を、継続的に大量のリクエストを送る洗練されたエージェントには「セッション型」を、といった使い分けが可能です。
- **複数方式の同時提示:** サーバーは 402 レスポンス内で複数の「チャレンジ」を返すことができます。クライアントは自身がサポートしている、あるいは最も効率が良い方式を選択して支払いを実行できます。

4. 開発効率の維持

MPPの設計は**加法的（Additive）**であるため、既存の一括払いルートにセッション型を追加する際、ルートハンドラー自体を大幅に変更する必要はありません。SDK（mppxなど）を使用すれば、どちらの方式で支払われたクレデンシャルであっても、サーバー側で自動的に検証・処理されます。

このように、一括払いによる**「手軽さ」**とセッション型による**「高速・安価な継続利用」**を組み合わせることで、多様なユーザー（マシンおよび人間）のニーズに応えることが可能になります。

SSEを使用したストリーミング支払いの仕組みは？

MPPにおける**SSE（Server-Sent Events）を使用したストリーミング支払い**は、高頻度な従量課金を行う「セッション型支払い」を拡張した仕組みです。LLM（大規模言語モデル）のトークン生成のように、内容を1語ずつ細かく配信しながら、その都度リアルタイムで課金を行う場合に適しています。

具体的な仕組みは以下の通りです。

1. セッションの確立とデポジット

ストリーミングを開始する前に、まず通常のセッション型支払いと同様の準備を行います。

- **初期リクエスト:** クライアントが有料のストリーミングエンドポイントにアクセスすると、サーバーは `402 Payment Required` と共に「デポジット（預託金）」を要求するチャレンジを返します。
- **チャネル開設:** クライアントが資金をデポジット（オンチェーンのエスクローへの預け入れや、Lightningでのインボイス支払い）することで、双方向の支払いチャネル（セッション）が開設されます。

2. ストリーミング中のリアルタイム課金

セッションが確立されると、サーバーはSSEを通じてデータの配信を開始します。

- **チャンクごとの課金:** サーバーは、例えば1トークンや1単語といった単位（チャンク）ごとに価格を設定し、データを送信するたびにセッション残高からその分を差し引きます。
- **高速な検証:** 各リクエストの検証は、サーバー側でのCPUによる署名チェック（ecrecoverなど）のみで行われるため、マイクロ秒単位の極めて低い遅延で処理されます。

3. 残高不足時の自動更新（ミッドストリーム更新）

ストリーミングの途中でデポジットした残高やバウチャーの限度額が不足した場合、**接続を維持したまま支払いを更新**できるのがこの仕組みの最大の特徴です。

- **通知イベント:** 残高がなくなると、サーバーはSSEのストリーム内で `payment-need-voucher`（Tempoの場合）または `payment-need-topup`（Lightningの場合）という特別なイベントを送信し、ストリームを一時停止します。
- **クライアントの自動対応:** クライアント側のSDK（`mppx`など）はこのイベントを検知すると、**ストリームを中断することなく**自動的に新しい支払い証明（バウチャーやトップアップ用のクレデンシャル）を署名・送信します。
- **再開:** サーバーが新しい証明を確認すると、ストリームは即座に再開されます。

4. セッションの終了と精算

ストリーミングが完了すると、セッションをクローズします。

- **最終精算:** 最終的な利用量に基づいた金額がサーバーに支払われ、未使用のデポジット（残高）がある場合はクライアントに返金されます。

この仕組みにより、サービス提供者は「1トークン単位」のような極小額の課金を、通信の遅延やネットワーク手数料を気にすることなく、安全かつ自律的に行うことが可能になります。

SSEストリーミングでの残高不足時の自動更新の流れは？

SSE（Server-Sent Events）を使用したストリーミング支払いにおいて、配信の途中で残高が不足した際の自動更新（ミッドストリーム更新）の流れは以下の通りです。

この仕組みにより、**接続を中断することなく**、リアルタイムで支払いを継続することが可能になります。

1. 残高不足の検知と配信の一時停止

サーバーはコンテンツ（LLMのトークンや単語など）をストリーミング配信しながら、1ユニットごとにチャネル残高から課金を行います。配信の途中で残高がなくなったことを検知すると、サーバーはデータの送信を一時停止し、接続を維持したままクライアントに通知を送ります。

2. 特定のSSEイベントによる通知

サーバーは、使用している決済ネットワークに応じて、ストリーム内で以下の特別なイベントを送信します。

- **Tempoの場合:** `payment-need-voucher` イベントを送信します。
- **Lightningの場合:** `payment-need-topup` イベントを送信します。

3. クライアントによる自動更新

クライアント側のSDK（`mppx`など）はこのイベントを検知すると、**ストリームを中断することなく**、自動的に新しい支払い証明（クレデンシャル）を作成して送信します。

- **Tempo:** クライアントが新しい「累積バウチャー」に署名し、サーバーに送信します。
- **Lightning:** クライアントが新しいデポジットインボイスを支払い、`topUp` クレデンシャルを送信します。

4. 配信の再開

サーバーが送られてきた新しい証明を検証すると、即座にチャネル残高が更新されます。検証完了後、サーバーは停止していた箇所からデータの配信を再開します。

この一連のプロセスはクライアントSDKによって**バックグラウンドで自動的に処理される**ため、ユーザーやエージェントはストリームが途切れることを意識せずにサービスを利用し続けることができます。

SSEストリーミングでの残高返金はどのように行われますか？

SSE（Server-Sent Events）を使用したストリーミング支払いにおける残高の返金は、**「セッション（ペイメントチャネル）をクローズする」**ことによって行われます。

ストリーミング支払いは高頻度な従量課金を行う「セッション」の仕組みに基づいているため、配信が完了した後に明示的にセッションを終了させることで、未使用の資金がクライアントに払い戻されます。

支払いネットワークごとの具体的な返金プロセスは以下の通りです。

1. Tempo（ステーブルコイン）の場合

Tempoでは、オンチェーンのエスクロー（預託）契約を使用して資金を管理します。

- **クローズの実行:** ストリーミング完了後、クライアント（またはサーバー）がセッションのクローズ（`session.close()`）を呼び出します。
- **精算と返金:** サーバーは受け取った中で「最も高い金額のバウチャー（支払い証明）」を使用してオンチェーンで精算を行います。この際、**エスクローにロックされていたデポジットから、実際に消費された分を差し引いた残額が、クライアントのウォレットへ自動的に返金**されます。

2. Lightning Network（ビットコイン）の場合

Lightningでは、セッション開始時にクライアントが提示した「返信用インボイス（return invoice）」が返金に使用されます。

- **クローズの実行:** クライアントが「クローズ・クレデンシャル」を送信します。
- **返金の計算:** サーバーは `返金額 = デポジット額 - 利用済額` を計算します。
- **支払いの実行:** **サーバーが、クライアントの返信用インボイスに対して未使用分のサトシ（ビットコインの最小単位）を支払う**ことで返金が完了します。

重要な注意点

- **自動クローズではない:** セッションは個別のリクエスト間で再利用するために開いたまま維持されるため、ストリーミングが終了しても自動的には閉じられません。未使用のデポジットを回収するには、**クライアント側で明示的にクローズ処理（****close()****）を実行する必要があります**。
- **レシートによる確認:** セッションをクローズすると、最終的な精算結果やオンチェーンの取引ハッシュ（`txHash`）を含むレシートが返されます。

このように、MPPはストリーミングのような「総額が事前に分からない」取引において、**最初にある程度の金額を預け、最後に使わなかった分を正確に手元に戻す**仕組みを提供しています。

Lightningネットワークでの手数料はどのくらいですか？

Lightningネットワークでの手数料は、使用する支払い形態（インテント）によって異なりますが、総じて**「マイクロペイメント（極小額決済）に適した非常に安価な設定」**となっています。

具体的な手数料の仕組みは以下の通りです。

1. 支払い形態による違い

- **一括払い（Charge）:** リクエストごとにLightningネットワーク上での送金（HTLCのルーティング）が発生するため、**毎回ネットワーク手数料がかかります**。クライアントSDKのデフォルト設定では、1回の支払いに対する最大手数料（`maxFeeSats`）は**100 satoshis**に制限されています。
- **セッション型（Session）:** 最初にまとめて資金をデポジット（預託）し、その後はオフチェーンの証明（プリイメージ）を使用してやり取りするため、**リクエストごとの手数料はほぼゼロ**になります。ネットワーク手数料が発生するのは、最初のデポジット時と、セッション終了後の残高返金時のみに集約されます。

2. 手数料の特徴

- **経済的なマイクロペイメント:** Lightningネットワークは**1セント（あるいは1円）未満の支払いも経済的にルーティングできる**ように設計されています。これにより、LLMの1トークン単位や1リクエスト単位といった、極めて単価の低いサービスへの課金が実用的になります。
- **コストの最小化:** 特に「セッション型」を利用することで、本来なら数百回のリクエストにかかるはずの手数料を、デポジットと精算の数回分に抑えることが可能です。

結論として、一括払いの場合はリクエストごとに少額のルーティング手数料がかかりますが、高頻度な利用が想定される場合は、セッション型を利用することで**リクエストあたりのコストを極限まで引き下げる**ことができます。

Tempoでの手数料肩代わり（Fee Sponsorship）とは？

Tempoでの**手数料肩代わり（Fee Sponsorship）****トランザクション手数料（ガス代）を支払う仕組み**のことです。

主な特徴と仕組みは以下の通りです。

1. 利用者のメリット

- **ガス代が不要:** クライアントは支払いのためのトークン（pathUSDなど）さえ持っていればよく、手数料支払いのためのネイティブトークン（ガス代）を保持する必要がありません。
- **UXの向上:** ユーザーやエージェントは複雑な「ガス代の仕組み」を意識することなく、純粋なサービスの利用料金のみを支払うことができます。

2. 技術的な仕組み

この機能は、Tempoの**「pull」モード**というトランザクション送信方式によって実現されます。

1. **クライアント:** 支払いの内容に署名（認証）し、そのデータをサーバーに送ります。
2. **サーバー:** 受け取った署名データに、サーバー自身の「手数料支払い者（fee payer）」としての署名を付け加えます（ドメイン 0x78 を使用）。
3. **実行:** サーバーが最終的なトランザクションをネットワークへブロードキャスト（放送）し、手数料はサーバーのアカウントから差し引かれます。

4. 対応している支払い形態

一括払い（Charge）とセッション型（Session）の両方で利用可能です。

- **一括払い:** リクエストごとの支払い手数料をサーバーが負担します。
- **セッション型:** チャネルの開設、追加入金（トップアップ）、クローズなどの管理手数料をサーバーが負担します。

4. サーバー側での設定方法

サーバー開発者は、`tempo()` メソッドの初期化時に `feePayer` パラメータを設定することでこの機能を有効にできます。

- **特定のアカウントで署名:** サーバーが保持するアカウントを直接指定します。
- **外部サービスへの委譲:** 外部の「手数料支払いサービス（fee payer service）」のURLを指定して処理を任せることも可能です。
- **簡易設定:** `feePayer: true` と設定することで、受け取り用アカウントを手数料支払い者として兼用させることもできます。

この仕組みにより、AIエージェントなどのマシンは、ウォレットの残高管理における複雑さを回避しながら、より自律的かつスムーズに有料サービスを利用できるようになります。
