# Quality Review Checklist: 5つの品質の柱

伝説のエンジニアブロガーとして、公開前に必ずチェックすべき5つの観点です。

## 1. 論理性 (Logicity)
構成に矛盾がなく、主張がスムーズにつながっているか。
- [ ] 導入で提示した問い（課題）に対して、結論で明確な答えが出ているか。
- [ ] 章の順序が、読者の思考プロセス（なぜ？→どうやって？→本当？）に沿っているか。
- [ ] 「AだからB」という論理に飛躍がないか。

## 2. 実用性 (Practicality)
読者が明日から使える具体的な知見やコードが含まれているか。
- [ ] コードはコピー＆ペーストで動く状態か（あるいは、動かすための前提条件が明記されているか）。
- [ ] 単なる「やり方」だけでなく、その技術を「いつ使うべきか（あるいは使うべきでないか）」が示されているか。
- [ ] 読者が陥りやすいエラーやハマりポイントへの言及があるか。

## 3. 読みやすさ (Readability)
リズムが良く、視覚的にも圧迫感のない文章か。
- [ ] 同じ語尾（〜です、〜です）が3回以上続いていないか。
- [ ] 適切な場所に箇条書きや図解（Mermaid）、メッセージボックスが配置されているか。
- [ ] 1つの段落が長すぎないか（スマホで見た時に5行を超えないのが理想）。

## 4. 独自性 (Uniqueness)
あなたにしか書けない視点や経験談が盛り込まれているか。
- [ ] 「公式ドキュメントの要約」以上の価値（著者の主観、失敗談、愛）があるか。
- [ ] 独自の比喩や、著者なりの「解釈」が含まれているか。
- [ ] AIが生成したような、誰にでも言える一般論に終始していないか。

## 5. 明確性 (Clarity)
結論や主張が曖昧にならず、はっきりと伝わるか。
- [ ] 専門用語が定義なしに使われていないか（必要に応じて注釈があるか）。
- [ ] 「〜だと思います」「〜かもしれません」といった曖昧な表現を、自信を持って言い切る形に書き換えられないか。
- [ ] 最も伝えたいメッセージ（One Message）が、記事全体を通して一貫しているか。
