# スマートコントラクトには質の高いコード監査が必須

現在、パブリックブロックチェーン上のスマートコントラクトというプログラムで **1,000億ドル（約15兆円）** を超える資産が管理されています。

![](/images/web3_evmbench-1/0.png)

有名どころだと**Uniswap**や**Aave**というDeFiがあり、多くの資産がプログラムによって管理されています。

https://app.uniswap.org

https://aave.com/

> DeFiとは
> 
> 分散型金融の略称。ブロックチェーン技術を活用し、管理者を介さずにスマートコントラクトによって自動的に運営される金融サービスのこと。

プログラムで管理されている以上、もし脆弱性があればその穴をハッカーに突かれて巨額の資産が流出することになります。

残念ながら過去に何度もDeFi系のプロジェクトで巨額の資産流出事故が発生してしまっています。

https://www.chainalysis.com/blog/2022-defi-hacks-japanese/

原因はヒューマンエラーなどコードの脆弱性のみに限りませんが、事前に対策しそびれてしまった脆弱性が理由で起きている事故がいくつもあります。

つい先日にはAIが生成したコードが原因で資産流失事故が発生したことも報じられたばかりで、これは記憶に新しいと思います。

https://crypto-times.jp/news-ai-code-flaw-causes-270-million-to-be-lost-from-defi/

**このような事故を起こさないためにブロックチェーンアプリケーションではコード監査が非常に重要**となっています。

さらに通常のプログラムとは異なるスマートコントラクト特有の性質として一度デプロイされてしまうとコードが全世界に公開されかつ**後からコードを修正することが非常に困難**という性質があります。

故に**脆弱性があるコードがデプロイされてしまうとそのまま悪用されてしまう可能性が非常に高くなります**。

だからデプロイ前のコード監査が非常に重要な意味を持ちます。

スマートコントラクトのコード監査に特化したサービスも存在しています。

https://quantstamp.com/

https://hiaudit.io/

# EVMbenchとは何なのか？

EVMbenchは、OpenAIと**Paradigm**が共同開発した、AIエージェントのスマートコントラクトセキュリティ能力を測定するためのベンチマークです。

https://www.paradigm.xyz/

> **Paradigm**はWeb3業界ではグローバルで著名な暗号資産投資企業です。

もう少し噛み砕くとAIがどのくらいスマートコントラクトの脆弱性を検知・修正・攻撃する能力を保有しているかどうかを測るためのデータセットとフレームワークになります。

上述した通り、**スマートコントラクトにとってデプロイ前のコード監査が非常に重要です**。

AIの発展が目覚ましい今、AIがどの程度スマートコントラクトの脆弱性を攻撃する能力を持っているのか理解しておくことは重要なことです。

そういった意味で今回のOpenAIの発表は大変重要な意味を持つと考えています。

# EVMbenchの3つの評価方法

スマートコントラクトの能力を計測する方法としてEVMbenchでは3つの評価方法が設定されていました。

## 1つ目：検知

「脆弱性を検出できるか」という指標ですね

- **内容:**   
  エージェントがスマートコントラクトのリポジトリを監査し、脆弱性を特定したレポートを作成します。

- **評価基準:**   
  人間の監査人が作成した「正解」との一致（再現率）と過去のコンペティション(後述するCode4area)に基づく「仮想報酬額」で採点されます。

![](/images/web3_evmbench-1/1.png)

ここで言及されている**Code4area**とはスマートコントラクトのコード監査をコンテスト形式で行っているプラットフォームです。

https://code4rena.com/

わかりやすくいうと**ホワイトハッカー向けのコンテスト**です。

私も過去に一度チャレンジしてみましたが、めっちゃむずかったです笑

その過去のコンテストがデータセット(120件)として用意されていました！

あとStripeが開発しているL1ブロックチェーン**Tempo**のコントラクト監査もデータセットに使われていました。

https://tempo.xyz/

以下のフォルダから確認ができます。

https://github.com/openai/frontier-evals/tree/main/project/evmbench

## 2つ目：修正

「安全なコードに修正できるか」という指標ですね

- **内容:**   
  脆弱なコードを直接編集し、バグを取り除きます。
- **評価基準:**  
  「既存の機能が維持されているか（テストのパス）」と「脆弱性が排除されたか（未公開の攻撃テストが失敗するか）」の2点で自動検証されます

![](/images/web3_evmbench-1/2.png)

## 3つ目：攻撃

最後に最も難しい「実際に脆弱性を利用して悪用できるか」という指標ですね

ただ単にコード書くだけでなく、**DeFiに対する深いコンテキストの理解が要求**されるので3つの中で一番難しいとされています。

- **内容:**   
  サンドボックス化されたブロックチェーン環境で、実際に資金を奪うエンドツーエンドの攻撃を実行します。
- **評価基準:**  
  送信されたトランザクションを再生し、チェーン上の残高変化やイベントをプログラムで確認して成功を判定します。

![](/images/web3_evmbench-1/3.png)

# EVMbenchの結果

## 評価対象のモデル一覧表

検出では **Claude Opus 4.6** がトップでそれ以外の2つの評価については **GPT-5.3-Codex**がトップだったようです。

| モデル名 | 開発元 | 特徴・備考 | 主なリザルト傾向 |
|---------|--------|----------|-----------------|
| GPT-5.3-Codex | OpenAI | 本ベンチマークの覇者。<br/>Codex CLI経由で実行。 | **Exploit成功率 72.2% で最高値を記録**。 |
| Claude Opus 4.6 | Anthropic | Claude Code経由で実行。 | **Detect（検出）モードで1位**。<br/>発見報酬額でトップ。 |
| GPT-5.2 | OpenAI | 高い推論能力を持つ<br/>xhigh reasoningモデル。 | 全モードでバランス良く高スコア。 |
| OC-GPT-5.2 | OpenAI | OpenCode（OC）スキャフォールド（枠組み）を使用。 | 標準モデルより修正や攻撃の精度が向上。 |
| Gemini 3 Pro | Google | Gemini CLI経由で実行。 | 検出・攻撃ともにトップ層に食い込む。 |
| Claude Opus 4.5 | Anthropic | 先代モデル。Opus 4.6以前の基準値。 | 4.6に比べ、自律的な攻撃継続力で劣る。 |
| GPT-5 | OpenAI | 2025年後半リリースのベースモデル。 | Exploit成功率は31.9%。5.3で劇的に進化。 |
| OpenAI o3 | OpenAI | 推論特化型モデル。 | 複雑なロジックの脆弱性検出に強み。 |


## 結果の総括

論文の中で各モードを横断的に評価した結果、AIは **「資金を奪う（Exploit）」という明確な目標があるタスクには長けているが、広大なコードから脆弱性を見つけ出す「検出」や機能を壊さず直す「修正」では苦戦する**ことが明らかになったと記載されています。

単なるコード生成や修正が課題ではなく、**「問題の所在を特定する発見フェーズ」** がボトルネックになっているみたいです。

でもこれはAIだけじゃなくて人間も同じですね笑。
