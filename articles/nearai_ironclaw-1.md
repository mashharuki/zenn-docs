# IronClawとは

![](/images/nearai_ironclaw-1/0.png)

**IronClaw**は、**Near AI**が開発したRust製のOSS AIエージェント・フレームワークです！

**NEAR AI Cloud**などのセキュアな環境で動作し、ユーザーのパスワードやAPIキーなどの機密情報をAI（LLM）に直接見せることなく、様々なツール（GoogleアプリやGitHubなど）を自律的に操作できる「自分専用のセキュアなAI Agent」を構築できるとされています！

## Near AI とは

https://www.near.org/ai

**Near AI**は、元々パブリックブロックチェーンでかつてはあの**Ethereum**にとって変わるブロックチェーンになるかもしれないとされていたプロジェクトでした！

今は方針転換してAIネイティブなパブリックブロックチェーンとなり、**IronClaw**のようなプロダクトを生み出しています！

## IronClawの概要・特徴

**IrowClaw**は**Near AI Cloud**上で実行させることで通常のOpenClawにはない以下のような恩恵を享受することができるようになっています！

- **TEE（信頼実行環境）**:   
    最大の特徴は、エージェントを**TEE（Trusted Execution Environment）** と呼ばれる暗号化された隔離空間（エンクレーブ）で実行させることができます。

- **データの秘匿性**:   
    メモリ内のデータは常に暗号化されており、クラウドプロバイダーであるNEAR AI自身であっても、ユーザーの会話内容やメモリ内の情報を覗き見ることはできません。

- **検証可能性**:   
    ハードウェアレベルの隔離（GPUアテステーションなど）により、プライベートで検証可能なコンピューティングを実現しています。

- **機密情報の保護**:  
    IronClawの「暗号化ボルト（Vault）」とNEAR AIのインフラが連携することで、AIに生のAPIキーを見せることなく外部ツールを操作できる安全なワークフローを実現しています。

この辺りが最大の特徴でしょう！

> セキュリティを担保するためにIronClawでは4層にもなる多層防御のアーキテクチャが採用されているのですがそれは後述します。

### 主要機能

OpenClawライクな実装になっているのでOpenClawにできることは一通りできます。

| カテゴリ | 機能名 | 内容とメリット |
| --- | --- | --- |
| セキュリティ | 暗号化Vault<br/>WASM隔離 | APIキーなどの秘密情報を暗号化して保管し、<br/>ツールはサンドボックス環境下で実行されます。 |
| 外部連携 | マルチチャネル対応 | ターミナルだけでなくTelegram、Discord、Webブラウザ、Webhook経由で操作可能です。 |
| 記憶 | 持続的メモリ | 会話の内容や文書を永続的に保存し、<br/>後からAIが検索して活用できます。 |
| 自動化 | ジョブ | 「毎日朝9時にレポートを作成する」といった定期実行や<br/>複数のタスクの同時並行処理が可能です。 |
| 拡張性 | ツール| Googleドキュメント、カレンダー<br/>GitHub、Web検索などの機能を自由に追加できます。 |

上記以外にもSKILLを追加したり、マークダウンファイルで設定を変えたり、MCPを接続したりすることが可能でした。


### IronClawの4層セキュリティアーキテクチャ

IronClawはデータが外部サービスに到達する前に、4つの独立した保護層を通過するようなアーキテクチャになっています！

> このようなアーキテクチャを採用することにより通常のOpenClawで発生したような機密情報の流失事故などを防ぐことができます！

1. セーフティレイヤー
プロンプトインジェクション防御
コンテンツのサニタイズ
ポリシーエンジン
リーク検出（15種類以上のシークレットパターンをスキャン） security.mdx:55-65
2. WASMサンドボックス
ツール実行のための軽量な隔離環境：

メモリ制限: デフォルト16MB sandboxed-tools.mdx:48-55
Fuelメータリング: 無限ループ防止（デフォルト1億命令） sandboxed-tools.mdx:39-46
機能ベースの権限: capabilities.jsonで明示的に宣言された機能のみ許可 sandboxed-tools.mdx:7
3. Dockerサンドボックス
ジョブ実行のためのコンテナ隔離：

オーケストレーター/ワーカーパターン
ネットワークプロキシ経由の通信
ジョブごとのトークン認証 sandbox.mdx:56-60
4. シークレット管理
ゼロエクスポージャーモデルを実現：

AES-256-GCM暗号化 index.mdx:14
OSキーチェーン連携
ホスト境界でのみ認証情報を注入（コンテナには渡らない） index.mdx:119-125

### WASMサンドボックスとDockerサンドボックスの違い

WASM（WebAssembly）サンドボックスとDockerはどちらもエージェントの動作を隔離するための重要なセキュリティ機能ですが、その仕組みや主な役割には明確な違いがあります。

主な違いを以下の表と詳細にまとめてみました！

| 項目 | WASMサンドボックス | Dockerサンドボックス |
| --- | --- | --- |
| 実行基盤 | wasmtime ランタイム | コンテナ型仮想化 |
| 主な用途 | カスタムツールの安全な実行 | コード実行、ビルド、信頼できない<br/>スクリプトの実行 |
| 軽量さ | 非常に軽量で高速に動作 | WASMに比べるとリソース消費が大きい |
| リソース制限 | フューエル・メータリング<br/>（命令単位での制限） | CPUやメモリ単位での制限 |
| 秘密情報の扱い | プロキシ経由で注入<br/>（ツールは生鍵を見ない） | 環境変数などの制限による保護 |
| 隔離の強さ | 非常に強力<br/>（OS機能へのアクセスが厳制限） | プロセスレベルの隔離 |
